Modelado de bases de datos

El modelado es el primer paso dentro de una implementación de base de datos. Es importante porque plasma los requerimientos de las aplicaciones y las reglas o restricciones que deben cumplir los datos.

Pasos para realizarlo:

Lo importante es tener en claro qué es lo que se tiene que almacenar, y es por ello que este requerimiento tiene que estar lo más detallado posible.

Una vez que tenemos el requerimiento, se inicia la etapa de modelado, que nos permite validar que entendimos el requerimiento y empezar a pensar cómo lo vamos a implementar.

Una vez implementada la base de datos, el modelado nos sirve de documento para volver a consultar y entender el porqué se implementó la base de datos de una determinada forma.

Modelado de datos

Un modelo es un conjunto de herramientas conceptuales para describir datos, sus relaciones, su significado y sus restricciones de consistencia.


El modelado de una base de datos es el proceso de crear un modelo de datos para los datos a almacenar en una base de datos. El modelo es una representación teórica que define la estructura de la base de datos y cómo se relacionan los datos entre síorganizar los datos para que se puedan utilizar fácilmente por las bases de datos.

Beneficios:

  • Registrar los requerimientos de datos de un proceso de negocio.
  • Se puede descomponer un proceso complejo en partes.
  • Permite observar patrones.
  • Sirve de plano para construir la base de datos física.
  • Ayuda a las empresas a comunicarse dentro y entre las organizaciones.
  • Proporciona soporte ante los cambios de requerimientos del negocio o aplicaciones.

Tipos de modelos

Existen tres tipos de modelos que podrían implementarse, y que pueden pensarse como distintas instancias, donde empezaremos con un modelado conceptual para entender al negocio, entrevistando al usuario para conocer los procesos que se realizan en la empresa.

Los datos obtenidos con el modelado conceptual se representan luego mediante un modelo lógico, que se centra en la estructura de los datos independientemente de su implementación física. En esta etapa se definen las entidades, los atributos de las entidades y las relaciones existentes.

Por ejemplo, si estamos diseñando una base de datos para una librería, las entidades podrían ser Libros, Autores y Pedidos. En el marco del modelo lógico especificamos qué datos sobre cada uno de los libros vamos a almacenar, o cómo se van a relacionar los libros con los autores. El modelo lógico se centra en determinar qué es lo que se almacenará y sin importar todavía cómo se almacenará.

La siguiente etapa es la del modelo físico, donde se toma como base el modelo lógico y se lo transforma en una implementación de base de datos de, por ejemplo, MySQL. Aquí se ven detalles técnicos como el tipo de datos de los atributos, índices, restricciones de integridad y la forma en que se almacenarán físicamente los datos: tablas, filas, columnas, etc. El modelo físico está vinculado con el rendimiento y eficiencia de la base de datos. Este proceso no es lineal porque si necesitamos modificar el modelo conceptual o lógico, también deberemos modificar el modelo físico. Este proceso iterativo permite asegurar que la base de datos sea funcional y eficiente.

Conceptual

Es un modelo con un diseño muy general y abstracto, cuyo objetivo es explicar la visión general del negocio o sistema.

Lógico

El modelo lógico es una versión completa que incluye todos los detalles acerca de los datos. Explica qué datos son importantes, su semántica, relaciones y restricciones. Explica el “qué”.

Físico

Es un modelo que implementa el modelo lógico. Es un esquema que se va a implementar dentro de un sistema de gestión de bases de datos. Explica el “cómo”.

Modelo Lógico

Un modelo de datos lógico describe los datos con el mayor detalle posible, independientemente de cómo se implementarán físicamente en la base de datos. Describe los elementos importantes del negocio, qué significa cada objeto, su nivel de detalle, cómo se relacionan entre sí y sus restricciones.

Características de un modelo lógico:

  • Se definen cuáles son los conceptos importantes sobre los que hay que almacenar información. Estos elementos se denominan entidades.
  • Se especifican todos los atributos para cada una de las entidades.
  • Se conectan las entidades mediante relaciones.
  • Se especifica la clave principal para cada entidad.
  • Se especifican las claves externas (claves que identifican la relación entre diferentes entidades).
  • La normalización ocurre en este nivel.

Modelo Físico

El modelo físico representa cómo se construirá el modelo en la base de datos. Un modelo físico de base de datos muestra todas las estructuras de tablas, incluidos el nombre de columna, el tipo de datos de columna, las restricciones de columna, la clave principal, la clave externa y las relaciones entre las tablas.

Características de un modelo físico:

  • Especificación de todas las tablas y columnas.
  • Las claves externas se usan para identificar relaciones entre tablas.
  • La desnormalización puede ocurrir según los requisitos del usuario.
  • Las consideraciones físicas pueden hacer que el modelo físico sea bastante diferente del modelo lógico.

El modelo físico puede diferir de un motor de bases de datos a otro —no es lo mismo implementar en SQL Server que en MySQL—.

Pasos para el diseño del modelo físico:

  • Convertir entidades en tablas.
  • Convertir relaciones en claves externas.
  • Convertir atributos en columnas.
  • Modificar el modelo de datos físicos en función de las restricciones / requisitos físicos.

MODELO LÓGICO

  • Describe el “qué”.
  • Explica el negocio.
  • Es independiente de la implementación.
  • Responsable: el analista.

MODELO FÍSICO

  • Describe el “cómo”.
  • Explica el técnicamente cómo se van a almacenar los datos.
  • Explica la implementación en el sistema de gestión de bases de datos.
  • Responsable: administrador de bases de datos o simil.

El modelado de bases de datos permite garantizar que la base de datos resultante responda a las necesidades del negocio.
